# ADR-004: Simple Pilot Operating Model

## Status

Accepted (final-close permission superseded by ADR-005)

## Date

2026-06-24

## Context

Phase 1 needs to support daily operation for a marmitaria pilot without becoming a broad restaurant management suite. The PRD leaves several implementation-level choices open around employee cost, cashier permissions, and expense categories. These choices affect the database schema, cash close calculation, and admin UI complexity.

## Decision

Phase 1 will use a simple pilot operating model:

- Each employee has a single `daily_cost_cents` used when the employee is marked present.
- Cashiers can open cash sessions and enter operational data, but only owners can final-close sessions and view full profit reports.
- Each pilot restaurant is seeded with basic marmitaria expense categories. The current Phase 1 default set is: Insumos, Infraestrutura, Manutenção, and Outros.

## Alternatives Considered

### Alternative 1: Flexible payroll model

- **Description**: Support daily, hourly, and monthly employee cost types from the start.
- **Pros**: Covers more employment arrangements.
- **Cons**: Adds form complexity and edge cases before the pilot proves the core workflow.
- **Why rejected**: Daily cost is enough to estimate personnel cost in the closing flow.

### Alternative 2: Cashier final close

- **Description**: Allow cashiers to final-close sessions while restricting reports.
- **Pros**: Reduces owner involvement in the daily workflow.
- **Cons**: Gives non-owner users control over sensitive final financial records.
- **Why rejected**: Owner confirmation is a better fit for pilot trust and permission simplicity.

### Alternative 3: No default expense categories

- **Description**: Require owners to create categories manually before recording expenses.
- **Pros**: Avoids assumptions about each restaurant.
- **Cons**: Adds setup friction and may slow pilot adoption.
- **Why rejected**: Basic marmitaria categories reduce first-use friction while still allowing an `other` bucket.

## Consequences

### Positive

- Keeps the Phase 1 schema and UI compact.
- Supports fast daily closing with fewer required decisions from staff.
- Gives the owner control over final profitability while still letting cashiers prepare data.

### Negative

- Payroll accuracy is approximate for employees with variable hours or monthly salaries.
- Expense analysis starts broad and may need refinement after pilot usage.

### Risks

- The daily-cost model may not match every employee arrangement; mitigate by allowing owners to edit `daily_cost_cents` before closing and by revisiting cost types after the pilot.
- Cashier inability to final-close may slow operation on days when the owner is unavailable; mitigate by making owner close review quick.

## Implementation Notes

Use `daily_cost_cents` as an integer-cent monetary field on employees, following [ADR-007](adr-007.md). Attendance records should snapshot the applicable daily cost for close calculation. Seed expense categories per restaurant during pilot provisioning. Role guards should permit cashier operational writes and final close (see ADR-005) but restrict adjustments, reports, employee cost visibility, and settings to owners.

## Amendment (2026-07-04)

The "Cashier final close" restriction described above and rejected under Alternative 2 was reversed: cashiers can now final-close a cash session without owner confirmation. The owner is not always present at closing time, and requiring confirmation blocked the daily close the pilot depends on. See [ADR-005](adr-005.md) for the full rationale; adjustments, reports, employee cost visibility, and settings remain owner-only as originally decided here.

The original category wording was also narrowed to match the PRD and TechSpec: Insumos, Infraestrutura, Manutenção, and Outros.
