# ADR-002: Seeded JWT Authentication for the Pilot

## Status

Accepted

## Date

2026-06-24

## Context

Phase 1 needs authenticated owner and cashier access with tenant isolation, but the PRD keeps self-serve onboarding and SaaS billing out of the MVP. The codebase is a greenfield NestJS and Next.js monorepo with no existing auth provider, no user model, and no tenant model. The pilot must validate daily cash closing for one real restaurant before investing in onboarding and invitation flows.

## Decision

Phase 1 will implement local email/password authentication in the NestJS API, seed the pilot restaurant and initial users in the database, and issue JWT access tokens containing `restaurant_id`, `user_id`, and role claims. API guards will derive tenant context from the JWT and enforce owner/cashier permissions. The admin app will provide a login screen but no self-serve signup or invitation flow in Phase 1.

## Alternatives Considered

### Alternative 1: Development-only manual JWTs

- **Description**: Skip login UI and use manually issued JWTs while building the API.
- **Pros**: Fastest backend bootstrap and fewer UI tasks.
- **Cons**: Does not test the pilot user's real access flow and delays role enforcement in the admin app.
- **Why rejected**: Phase 1 needs owner and cashier users to operate the product daily without developer intervention.

### Alternative 2: Full onboarding and invitations

- **Description**: Build restaurant signup, owner onboarding, and staff invitation flows in Phase 1.
- **Pros**: Moves closer to SaaS self-serve behavior.
- **Cons**: Expands scope into Phase 3 concerns and distracts from cash-first validation.
- **Why rejected**: The pilot can be provisioned manually, and self-serve onboarding is explicitly out of MVP scope.

### Alternative 3: External authentication provider

- **Description**: Use a hosted auth provider for credentials, sessions, and user management.
- **Pros**: Reduces custom credential handling and may simplify future integrations.
- **Cons**: Adds vendor dependency and integration complexity before the core pilot workflow is validated.
- **Why rejected**: Local credentials are sufficient for seeded pilot users and keep Phase 1 operationally simple.

## Consequences

### Positive

- Keeps authentication real enough for the pilot without building self-serve SaaS flows.
- Establishes the JWT claims needed for tenant scoping and role-based guards.
- Allows the admin app to exercise owner and cashier experiences from the first vertical slice.

### Negative

- User provisioning remains manual during Phase 1.
- Password reset, invitations, and account lifecycle management are deferred.

### Risks

- Local credential handling must be implemented carefully; mitigate with password hashing, strict validation, and no plaintext secrets in seed data.
- Future self-serve onboarding may require evolving seeded provisioning into automated restaurant creation.

## Implementation Notes

Use Prisma models for `Restaurant`, `User`, and role membership. Add NestJS auth guards for JWT validation and role checks. Seed one pilot restaurant, one owner, and at least one cashier. The frontend should store tokens using the project-approved session approach during implementation and redirect unauthenticated users to login.
